# FindAm Backend - AI Development Guide

This document provides detailed guidance for Claude and Cursor AI when working with the FindAm project.

## Project Overview

**FindAm Backend** is a Rails 8.0 GraphQL API backend for a property rental platform connecting landlords and tenants. It's a greenfield project implementing Single Table Inheritance (STI) for user polymorphism with comprehensive GraphQL support.

**Key Statistics:**

- Rails 8.0 with Ruby 3.4
- PostgreSQL database
- Pure GraphQL API (no traditional REST endpoints)
- Devise authentication via GraphQL Devise (version ~2.1)
- Real-time subscriptions via Action Cable
- Background jobs: Solid Queue
- Caching: Solid Cache
- WebSocket: Solid Cable

**Database:** `uniguideme_backend_development`

## Architecture & Design Patterns

### 1. Single Table Inheritance (STI) for Users

The application uses STI with a base `User` model and three subclasses:

```
User (base)
├── Landlord
├── Tenant
└── Admin
```

**Key Files:**

- `app/models/user.rb` - Base User model with Devise authentication
- `app/models/landlord.rb` - Landlord STI subclass
- `app/models/tenant.rb` - Tenant STI subclass
- `app/models/admin.rb` - Admin STI subclass

**User Columns (from schema):**

- `type` - STRING (stores "User", "Landlord", "Tenant", or "Admin")
- `email` - STRING, unique
- `encrypted_password` - STRING
- `provider` - STRING (default: "email")
- `uid` - STRING (unique with provider)
- `onboarding_completed` - BOOLEAN (default: false)
- `onboarding_completed_at` - DATETIME
- `approval_status` - STRING ("pending", "submitted", "approved", "rejected")
- `approved_at` - DATETIME
- `rejected_at` - DATETIME
- `rejection_reason` - STRING
- `confirmation_token`, `confirmed_at`, `reset_password_token` - Devise fields
- `tokens` - JSON (authentication tokens)

**User Methods:**

```ruby
user.landlord?      # => true/false
user.tenant?        # => true/false
user.admin?         # => true/false
user.approve!       # Mark as approved
user.reject!(reason) # Mark as rejected
```

### 2. Profile Models

Each user has an associated profile (polymorphic via profile_type):

- `Profile` (base) - Basic user profile
- `LandlordProfile` - Landlord-specific details
- `TenantProfile` - Tenant-specific details

**Relationship:**

```ruby
User has_one :profile (dependent: :destroy)
User has_one :landlord_profile (as Landlord)
User has_one :tenant_profile (as Tenant)
```

Profiles are **automatically created** upon user creation via `after_create :create_profile`.

## GraphQL API Structure

### Endpoint

- **POST** `/graphql` - Main GraphQL endpoint
- **GET** `/graphiql` - GraphQL IDE (development only)
- **POST** `/graphql_auth/*` - Devise authentication endpoints

### Query Flow

**QueryType** (`app/graphql/types/query_type.rb`):

```graphql
query {
  currentUser { /* Returns authenticated user */ }
  node(id: "ID") { /* Relay node interface */ }
  nodes(ids: ["ID"]) { /* Batch node fetching */ }
  testField { /* Example field */ }
}
```

**MutationType** (`app/graphql/types/mutation_type.rb`):

- `userRegistration` - Register new user (Landlord, Tenant, Admin)
- `completeStudentOnboarding` - Complete onboarding process

### Authentication Model

**GraphQL Devise Integration:**

- Built-in mutations provided: `login`, `logout`, `register`, `updatePasswordWithToken`, etc.
- Not used in favor of custom `userRegistration` mutation
- Token-based auth via JWT in headers
- Authentication via `context[:current_resource]`

**Custom Registration Flow:**

```ruby
# Mutation: userRegistration
userRegistration(
  email: "user@example.com",
  password: "secret123",
  passwordConfirmation: "secret123",
  type: "LANDLORD"  # or "TENANT", "ADMIN"
) {
  authenticatable { id email type }
  credentials { auth_header access_token }
}
```

**GraphQL Types** (`app/graphql/types/`):

**UserType Exposes:**

- All authentication fields (email, encrypted_password, etc.)
- Profile relationships (landlord_profile, tenant_profile)
- Approval workflow (approval_status, approved_at, rejection_reason)
- Onboarding state (onboarding_completed, onboarding_completed_at)

### Mutation Pattern

All mutations inherit from `Mutations::BaseMutation`:

```ruby
class Mutations::BaseMutation < GraphQL::Schema::RelayClassicMutation
  argument_class Types::BaseArgument
  field_class Types::BaseField
  input_object_class Types::BaseInputObject
  object_class Types::BaseObject
end

# Custom mutation example
class Mutations::UserRegistration < BaseMutation
  argument :email, String, required: true
  argument :type, Types::Enums::UserTypeEnumType, required: true

  field :authenticatable, Types::Objects::UserType, null: true
  field :credentials, GraphqlDevise::Types::CredentialType, null: true

  def resolve(email:, password:, type:)
    # Implementation
  end
end
```

## Key Implementation Details

### 1. User Registration

**File:** `app/graphql/mutations/user_registration.rb`

**Process:**

1. Creates user with specified type (Landlord, Tenant, Admin)
2. Auto-creates associated profile
3. Returns authentication credentials
4. Email confirmation is disabled (TODO: may be re-enabled)

### 2. Onboarding Workflow

**File:** `app/graphql/mutations/complete_onboarding.rb`

**Process:**

1. Updates `onboarding_completed = true`
2. Sets `onboarding_completed_at` to current time
3. Changes `approval_status` to "submitted"
4. Requires authentication

**Flow:**

```
User Created (approval_status: "pending")
    ↓
Complete Onboarding (approval_status: "submitted")
    ↓
Admin Review
    ├─→ approve! (approval_status: "approved")
    └─→ reject! (approval_status: "rejected")
```

### 3. Approval Workflow

**Methods in User Model:**

```ruby
user.approve!
# Sets: approval_status = "approved", approved_at = Time.current

user.reject!(rejection_reason: "Reason text")
# Sets: approval_status = "rejected", rejected_at = Time.current, rejection_reason = "..."
```

## Conventions & Patterns

### Naming Conventions

- **GraphQL Types:** `Types::Objects::UserType` (singular, CamelCase)
- **GraphQL Enums:** `Types::Enums::UserTypeEnumType` (ends with "EnumType")
- **GraphQL Mutations:** `Mutations::UserRegistration` (verb-noun)
- **Models:** Singular (User, Profile, Landlord)
- **Database Tables:** Plural (users, profiles, landlord_profiles)

### GraphQL Field Naming

- Field names in GraphQL are `camelCase`
- Ruby method names are `snake_case`
- Mapping happens automatically via GraphQL

Example:

```ruby
field :onboarding_completed, Boolean  # Exposed as "onboardingCompleted" in GraphQL
field :approval_status, String        # Exposed as "approvalStatus" in GraphQL
```

### Error Handling

Use `GraphQL::ExecutionError` for mutation errors:

```ruby
raise GraphQL::ExecutionError.new(
  "Error message",
  extensions: model.errors.to_hash
)
```

### Base Classes

Always extend from base classes:

- Mutations: `Mutations::BaseMutation`
- Types: `Types::BaseObject`, `Types::BaseScalar`, etc.
- Resolvers: `Resolvers::BaseResolver` (if used)

## Development Workflows

### Adding a New Mutation

1. Create file: `app/graphql/mutations/mutation_name.rb`
2. Extend `Mutations::BaseMutation`
3. Define arguments and return fields
4. Implement `resolve` method
5. Register in `Types::MutationType`

Example:

```ruby
class Mutations::UpdateUserProfile < BaseMutation
  argument :name, String, required: true
  field :user, Types::Objects::UserType, null: true
  field :errors, [String], null: false

  def resolve(name:)
    current_user.update(name: name)
    { user: current_user, errors: [] }
  end
end
```

Then add to `mutation_type.rb`:

```ruby
field :update_user_profile, mutation: Mutations::UpdateUserProfile
```

### Adding New Fields to Types

1. Open type file (e.g., `app/graphql/types/objects/user_type.rb`)
2. Add field with resolver if needed
3. Use `null: false` for required fields

```ruby
field :custom_field, String, null: false
field :related_object, Types::Objects::RelatedType, null: true

def custom_field
  object.some_method
end
```

### Testing User Types

Use RSpec with Factory Bot:

```ruby
describe "User Registration" do
  let(:query) { # GraphQL query string }

  it "registers a landlord" do
    result = FindamSchema.execute(query, variables: { type: "LANDLORD" })
    expect(result["data"]["userRegistration"]["authenticatable"]["type"])
      .to eq("Landlord")
  end
end
```

## Important Notes for AI Assistants

### Do NOT Do:

1. **Don't modify GraphQL Devise configs directly**
   - The resource loader is already configured for User
   - Custom mutations are preferred over framework mutations

2. **Don't delete or modify `type` column**
   - Core to STI pattern
   - Required for polymorphism

3. **Don't use REST endpoints**
   - This is a pure GraphQL API
   - All interactions go through `/graphql`

4. **Don't bypass authentication middleware**
   - Mutations should check `current_user` via `context[:current_resource]`
   - Use `authenticate: false` only for public mutations

5. **Don't modify Devise configuration lightly**
   - Currently configured with: auth, registration, recovery, rememberable, confirmable
   - Email confirmation code is disabled (TODOs in codebase)

### Do's:

1. **Always use Relay mutations pattern** for consistency
2. **Validate inputs in mutation arguments**, not resolve method
3. **Return errors as field in mutation response**, not just exceptions
4. **Create profiles through model callbacks**, not manually
5. **Use base classes** for all GraphQL definitions
6. **Document complex resolver logic** with inline comments
7. **Use `null: false` for required fields** to enforce schema contracts
8. **Test mutations with fixtures/factories** before integration

## Common Debugging Scenarios

### Issue: User type not recognized in GraphQL

**Solution:** Ensure `type` column in users table contains exact class name ("Landlord", "Tenant", etc.)

### Issue: Authentication context is nil

**Solution:** Check that GraphQL Devise is properly configured; middleware should set `context[:current_resource]`

### Issue: Profile not auto-creating

**Solution:** Verify `after_create :create_profile` callback is present and profile_type logic is correct

### Issue: GraphQL field camelCase

**Solution:** This is expected behavior. GraphQL auto-converts snake_case Ruby to camelCase queries
